ADC — Analog to Digital Converter

The bridge between the real physical world (continuous voltages from sensors) and the digital world inside your ECU (discrete numbers that software can process).

In one sentence: An ADC takes a continuous voltage (say 0–5V from a temperature sensor) and converts it into a digital number (say 0–4095) so that the ECU's microcontroller and AUTOSAR software can read, process, and act on it.
The basics

What goes in, what comes out

A sensor in the vehicle — a temperature sensor on the coolant pipe, a pressure sensor in the brake line, a throttle position sensor on the pedal — produces a continuous analog voltage. This voltage changes smoothly over a range, for example 0V to 5V. The hotter the coolant, the higher the voltage. That's the input.

The ADC samples this voltage at a specific moment in time and converts it into a binary number. If your ADC has 12-bit resolution (very common in automotive), it maps the 0–5V range into 0–4095 discrete steps. So 2.5V becomes roughly 2048. That number is what the software works with. That's the output.

Input (Analog side)

Continuous voltage from a sensor, typically 0–5V or 0–3.3V. Changes smoothly — not steps.

Output (Digital side)

A fixed-width binary number (e.g. 12-bit = 0 to 4095). Discrete steps. This is what software reads.

Resolution

Bit depth determines precision. 10-bit = 1024 steps. 12-bit = 4096 steps. More bits = finer measurement.

Sampling rate

How many times per second the ADC reads the voltage. Fast-changing signals need higher rates.

AUTOSAR layer stack — ADC signal path

Layer-by-layer breakdown

  • ADC Hardware Register: The microcontroller's ADC peripheral writes the conversion result (a raw integer like 1746) into a specific memory address. This is pure hardware — no software logic yet.
  • MCAL ADC Driver: The lowest software layer. It knows the specific microcontroller's register addresses. It provides a standardized AUTOSAR API (Adc_ReadGroup()) so upper layers don't need to know which chip they're running on.
  • IoHwAb (I/O Hardware Abstraction): This is where the raw number becomes meaningful. It applies a conversion formula or lookup table to translate 1746 into 87.3°C. After this layer, the value has engineering units.
  • RTE (Runtime Environment): The post office of AUTOSAR. Takes the value from IoHwAb's output port and delivers it to whichever Application SWC subscribed to it, using a Sender-Receiver interface.
  • Application SWC: Your actual vehicle function logic. Reads CoolantTemp = 87.3°C and decides what to do — turn on the cooling fan, adjust fuel injection, trigger a warning lamp.

Watch how the data transforms at each stage — from a physical voltage to a fan switching on:

Data transformation at each stage

The sensor (an NTC thermistor) changes its resistance with temperature. A voltage divider circuit converts that resistance into 2.13V. The ADC on channel AN3 produces the integer 1746 (because 2.13 / 5.0 × 4095 ≈ 1746). The MCAL ADC driver reads this. The IoHwAb applies a calibration curve to get 87.3°C. The RTE delivers it to the Engine Cooling SWC, which compares it against a threshold and turns the fan on.

Your role

Why this matters for a safety manager

The ADC is a single point of failure in many safety-critical signal chains. If the ADC gives a wrong reading, the software will make a wrong decision based on perfectly correct logic — it just has bad data.

ISO 26262 safety relevance

For any safety-related sensor signal (braking pressure, steering angle, throttle position), you need to analyze: what happens if the ADC reads too high? Too low? Stuck at zero? Stuck at max? These are your failure modes, and each needs a safety mechanism — such as range checking, redundant ADC channels, or plausibility checks.

Common ADC failure modes

Stuck-at-zero, stuck-at-max, offset drift, gain error, missing samples, cross-talk between channels, bit flips.

Typical safety mechanisms

Redundant channels, range monitoring, plausibility checks, ADC self-test (built into MCU), end-to-end signal protection.

ASIL impact

If the sensor signal is ASIL-rated, the entire chain (sensor → wiring → ADC → MCAL → IoHwAb → SWC) inherits that ASIL.

Hardware metrics

ADC contributes to SPFM and LFM calculations for the ECU hardware safety case.

Meeting survival

Terms you'll hear in meetings about ADC

  • "ADC group" — A set of ADC channels converted together in one trigger. Your software team configures which channels belong to which group.
  • "Conversion time" — How long one ADC sample takes (typically 1–2 μs). Matters when you have many channels and tight timing.
  • "ADC notification" — A callback function that AUTOSAR calls when a conversion group finishes. This is how software knows fresh data is ready.
  • "Raw value vs physical value" — Raw = integer from the ADC register. Physical = engineering unit after conversion (°C, kPa, V). IoHwAb does this translation.
  • "Scaling" or "calibration curve" — The formula or lookup table that converts raw ADC values to physical units. Often provided by the sensor supplier.
  • "ADC self-test" — Built-in hardware diagnostic where the ADC converts a known reference voltage and checks the result. Critical for ASIL-rated signals.
  • "Channel cross-talk" — When reading one ADC channel is affected by voltage on an adjacent channel. A hardware layout issue.
  • "Sample and hold" — The ADC captures the voltage at one instant and holds it steady during conversion. Fast signals need careful timing.
Physical location

Where is the ADC on an actual ECU?

The ADC is not a separate chip on the circuit board (in most automotive ECUs). It is built into the microcontroller itself. Microcontrollers like the Infineon AURIX, Renesas RH850, or NXP S32K come with multiple ADC units on the silicon die.

On the physical board, input pins (labeled AN0, AN1, AN2, etc.) connect from the board's connector (where sensor wires plug in) through signal conditioning circuitry (resistors, capacitors) into the microcontroller's ADC input pins.

Typical automotive MCU ADC specs

  • 12-bit resolution (4096 steps) — some newer MCUs offer 14-bit
  • Multiple ADC units (e.g. AURIX has 8+ modules) for parallel conversion
  • 16 to 48+ channels per unit — each channel maps to one physical pin
  • Conversion time around 1–2 microseconds per sample
  • Built-in: DMA transfer, hardware triggering, result FIFO buffers
  • Self-test and diagnostic capabilities for ISO 26262 compliance
