AUTOSAR OS Scalability Class 3 (SC3)

The OS configuration that adds memory fences between software components — so a bug in one piece of code can't silently corrupt the data of another. This is how you get "freedom from interference" on a single ECU.

In one sentence: SC3 takes the basic AUTOSAR OS (SC1) and adds Memory Protection, OS-Applications, Service Protection, and Trusted Functions — allowing you to run safety-critical and non-safety code on the same microcontroller while proving to ISO 26262 that one cannot interfere with the other.
The basics

What is it, and why does it exist?

In the AUTOSAR standard, the Operating System (OS) isn't one-size-fits-all. It comes in four Scalability Classes (SC1 through SC4), each adding safety and protection features on top of the previous one. Think of them like trim levels on a car — SC1 is the base model, SC4 is fully loaded.

Scalability Class 3 (SC3) is the one that focuses on spatial protection — keeping software components isolated in their own memory regions so they can't corrupt each other. It builds on SC1 (which already gives you scheduling, alarms, schedule tables, and stack monitoring) and adds four critical capabilities:

Memory Protection

Uses the MCU's hardware Memory Protection Unit (MPU) to enforce boundaries. Each OS-Application gets its own memory region. Write outside it → hardware exception → OS catches it.

OS-Applications

Groups of OS objects (Tasks, ISRs, Alarms, Counters, Schedule Tables) bundled into a logical unit. Each can be "Trusted" (full access) or "Non-Trusted" (restricted). This is the partitioning unit.

Service Protection

Validates every OS API call at runtime. A Non-Trusted task can't terminate a Trusted task, can't disable interrupts beyond its budget, and can't call APIs it doesn't have permissions for.

Trusted Functions

A controlled gateway so Non-Trusted code can safely call privileged operations. The OS switches to privileged mode, runs the function, switches back. Like a supervised bank vault door.

Input / Output

What goes in, what comes out

The AUTOSAR OS is a real-time operating system (RTOS) — it manages which piece of code runs when, and for how long. SC3 doesn't change what the OS fundamentally does (schedule tasks). Instead, it adds a protection wrapper around every task execution. Here's how to think about the inputs and outputs:

Inputs (configuration)

OIL/ARXML configuration defines: which tasks exist, their priorities, which OS-Application they belong to, whether each OS-Application is Trusted or Non-Trusted, and MPU region definitions.

Inputs (runtime)

Task activations from alarms, schedule tables, or the RTE. Interrupt requests from hardware. API calls from application software components (SWCs) and BSW modules.

Outputs (normal)

Scheduled execution of tasks in priority order. Runnable entities executed at the right time. Data passed between SWCs via the RTE. The application works as designed.

Outputs (protection event)

When a violation occurs: hardware exception → ProtectionHook() called → OS can terminate the offending OS-Application, restart it, or shut down the entire OS. This is logged via DEM.

The AUTOSAR OS is part of the Service Layer in the Basic Software (BSW). It has special permission to reach down and touch the hardware directly (the "L-shaped" access pattern in AUTOSAR). SC3 specifically leverages the MCU's Memory Protection Unit (MPU) — a piece of hardware inside the microcontroller.

AUTOSAR layer stack — SC3 protection boundaries

Notice the red "MPU WALL" dividers between OS-Applications at the top. These represent real hardware-enforced boundaries: the MPU inside the microcontroller is reprogrammed by the OS at every task switch to only allow memory access within the current OS-Application's assigned region. If a runnable in OS-Application A tries to write to memory belonging to OS-Application B, the MPU raises an exception before the write completes.

Key concept deep-dive

Trusted vs Non-Trusted OS-Applications

This is the most important concept for a safety manager to understand about SC3. Every OS-Application is classified as either Trusted or Non-Trusted, and this classification determines everything about how the code runs:

Non-Trusted OS-Applications (the restricted ones)

  • Run in the MCU's User Mode (non-privileged). Cannot directly access hardware registers or other partitions' memory.
  • Every memory access is checked by the MPU in hardware. Writing outside their region triggers an exception immediately.
  • Every OS API call is validated by Service Protection. Can't call TerminateApplication() on another OS-Application.
  • To access privileged operations, must use the Trusted Function mechanism — the OS briefly switches to privileged mode, runs the function, and switches back.
  • This is where your QM (non-safety) or lower-ASIL software components typically live.

Trusted OS-Applications (the privileged ones)

  • Run in the MCU's Supervisor Mode (privileged). Can access hardware directly and call any OS API.
  • Not subject to memory protection or service protection enforcement at runtime (they are "trusted" to behave correctly).
  • The BSW modules (including the OS itself) typically run as part of a Trusted OS-Application.
  • High-ASIL application SWCs can also be placed here — but this means they must be developed and verified to the highest ASIL level.
  • A bug in Trusted code can still corrupt anything. Trust is earned through rigorous development process, not runtime enforcement.
⚠ ISO 26262 implication

Memory protection (SC3) is the primary technical measure for demonstrating "freedom from interference" (ISO 26262 Part 6, Clause 7.4.4). Without it, a bug in a QM software component could corrupt an ASIL-D safety function — and you'd have to develop the QM component to ASIL-D standards (very expensive). With SC3, you can argue that the MPU hardware prevents this interference, allowing true mixed-ASIL deployment on a single ECU.

Why this matters for a safety manager

SC3 and ISO 26262

As a safety manager, SC3 is one of the most important architectural decisions on any ECU project. It directly impacts your safety case, your FMEA, your DFA (Dependent Failure Analysis), and your ability to mix different ASIL levels on one chip.

Freedom from interference

ISO 26262 Part 6 §7.4.4 requires demonstrating that software components of different ASIL ratings don't interfere with each other. SC3's MPU-based memory protection is the primary technical evidence for this.

Mixed-ASIL ECU

Without SC3: QM code on the same ECU as ASIL-D code must be developed to ASIL-D. With SC3: QM code stays QM because the MPU prevents it from affecting safety code. Massive cost savings.

Failure detection

SC3 detects memory corruption attempts in real-time via the ProtectionHook(). This is a diagnostic safety mechanism that directly improves your Diagnostic Coverage (DC) figures.

Partition restart

When a violation is detected, the OS can terminate and restart just the offending OS-Application — not the whole ECU. This is a "graceful degradation" strategy that auditors appreciate.

⚠ Key failure modes SC3 protects against

Wild pointer writes: A bug in QM code causes a pointer to write to an arbitrary memory address that happens to be in the safety-critical partition. Without SC3, the safety function silently receives corrupted data and makes a wrong decision. With SC3, the MPU catches the illegal write and triggers the ProtectionHook before any damage occurs.

Stack overflow across partitions: A task's stack overflows into another partition's data region. SC3's MPU regions detect this and raise an exception. (Note: basic stack monitoring in SC1 only checks a marker — SC3 uses hardware enforcement.)

Runaway code execution: Corrupted code jumps to a memory region it shouldn't execute. The MPU can also enforce execute permissions, preventing code injection across partitions.

What SC3 does NOT protect against

  • Timing failures — A task that runs too long, starving other tasks. That requires SC2's timing protection (or SC4 for both).
  • Bugs within the same partition — If two SWCs are in the same OS-Application, they share memory. The MPU doesn't protect them from each other.
  • Bugs in Trusted code — Trusted OS-Applications bypass the MPU. A bug in BSW or trusted SWC code can still corrupt anything.
  • Hardware common-cause failures — If the MCU core itself fails, the MPU (which is on the same die) fails too. This is a dependent failure that needs separate analysis.
Physical location

Where is SC3 on the actual ECU hardware?

The AUTOSAR OS (including SC3 features) is software running on the MCU — there is no separate "OS chip." However, SC3 relies critically on a specific piece of hardware inside the microcontroller: the Memory Protection Unit (MPU).

The MPU is a small hardware block inside the processor core itself. It sits between the CPU and the memory bus. Every time the CPU reads or writes memory, the MPU checks the address against its configured regions. If the access is not allowed, the MPU raises a hardware exception — faster than any software check could do.

MPU details on common automotive MCUs

  • Infineon AURIX TC3xx: Up to 16 data protection ranges and 10 code protection ranges per core. Supports privileged/user mode. The go-to MCU for SC3/SC4.
  • Renesas RH850: MPU with 8–16 configurable regions. Supports supervisor/user mode switching.
  • NXP S32K3: ARM Cortex-M7 based, with ARM MPU providing 8–16 regions. Requires careful configuration for AUTOSAR partition mapping.
  • Region count matters: Each OS-Application needs at least 2–3 MPU regions (code, data, stack). With limited regions, you can't have too many separate partitions.

Meeting survival

Terms you'll hear in meetings about OS SC3

  • "Partition" — Shorthand for OS-Application. When someone says "we have three partitions," they mean three OS-Applications with MPU-enforced boundaries.
  • "MPU region" / "protection region" — A configurable address range in the hardware MPU. Each has read/write/execute permissions. The OS loads the correct set of regions on each task switch.
  • "Trusted" / "Non-Trusted" — Whether an OS-Application runs in privileged (supervisor) or restricted (user) CPU mode. Trusted = no runtime checking. Non-Trusted = everything is checked.
  • "ProtectionHook" — The OS callback function invoked when any protection violation occurs (memory, service, or timing in SC4). You decide the reaction in this hook.
  • "IOC" (Inter-OS-Application Communicator) — The mechanism for passing data between OS-Applications. Generated by the RTE generator. Think of it as a mailbox that the OS controls.
  • "Freedom from interference" / "FFI" — The ISO 26262 requirement that software of different ASIL levels cannot affect each other. SC3 is the primary technical argument for FFI.
  • "CallTrustedFunction()" — The OS API that Non-Trusted code uses to execute a function in privileged mode. The OS switches CPU mode, runs the function, and switches back.
  • "Mixed-criticality" / "mixed-ASIL" — Running software components of different safety levels (QM, ASIL-A, ASIL-D) on the same ECU. SC3 is what makes this architecturally defensible.
  • "User mode" / "Supervisor mode" — CPU privilege levels. Non-Trusted apps run in user mode (restricted). Trusted apps and the OS run in supervisor mode (full access). Hardware-enforced.
  • "DaVinci Configurator" — Vector's tool for configuring the entire AUTOSAR BSW stack, including OS-Application assignments, MPU regions, and protection settings.
